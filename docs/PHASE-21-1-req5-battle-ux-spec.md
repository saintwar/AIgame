# 需求5 搏斗 UX 设计 · 「信息上鱼身」方案

> 版本：UX-spec v1.0
> 日期：2026-06-03
> 老板拍板方向：「搏斗时，右下角的 HP 和拉力值，都显示在鱼的身体下方。」
> 老板原始痛点：「水下和鱼搏斗时，右下角的鱼HP和鱼线拉力，会分散用户的视觉，如何处理会让用户更专注的看鱼的状态和控制拉力呢？」
> 交付对象：主程（CodeBuddy）。协作：本文设计 → 主程实施。
> 锚点基线：已核 `js/fishing-scene.js` `_renderPlaying()`（L2341 起）真实代码，下方所有改法对得上仓库。

## 0. 核资产发现 ——「现状不是右下角，而是 HP 显示了两份」（这是设计的关键前提，请老板先看这条）

核仓后发现一个老板体感之外的真相，它直接改变方案性质：

**当前 `_renderPlaying()` 里 HP 其实已经画在鱼身下方了，右下角木牌是第二份重复显示：**

| 信息 | 位置1（已在鱼身下方） | 位置2（右下角木牌面板） |
|---|---|---|
| HP 条 | ✅ `fishHPBarY = playingFishY + 55`，宽120，文字「X/Y」（**L2467-2473，已跟随鱼身**） | ✅ `hpBarY = panelY + 60`，文字「鱼的体力 X/Y」（L2537-2539） |
| 拉力条 | ❌ 无 | ✅ `tensionBarY = panelY + 100`，含黄金区高亮+▲指针，文字「鱼线拉力 X/Y」（L2542-2595） |

**结论**：老板说的"右下角分散注意力"，本质是**右下角那块木牌面板（HP+拉力一整块）和鱼身下方已有的 HP 条形成了重复 + 视觉分流**。所以需求5 不是"把信息从 A 平移到 B"，而是：

> **① 删掉右下角木牌面板这份重复占位；② HP 条统一只保留鱼身下方那份（已存在，微调）；③ 拉力条是唯一真正需要"搬家"的信息，但它恰恰是最不适合无脑跟随鱼身的——这正是老板痛点的核心，下文 §2 重点设计。**

这个发现让方案从"平移工作量"变成了"去重 + 拉力专项设计"，且更贴合老板"让玩家专注看鱼"的目标。

## 1. 布局方案：鱼身状态簇

### 1.1 整体形态
鱼身下方组成一个**紧凑竖向状态簇**，以 `playingFishX/playingFishY` 为锚：

```
        🐟  (playingFishX, playingFishY)
        ▔▔▔▔▔▔▔▔
   [████████░░]   ← HP 条（已存在）  Y偏移 +55
       58/130
   [██░░░░░░░░]   ← 拉力条（缩窄版）  Y偏移 +78
```

### 1.2 相对偏移与尺寸（基于现有 `playingFishX/Y`）
| 元素 | X 锚 | Y 偏移 | 宽 × 高 | 来源 |
|---|---|---|---|---|
| HP 条 | `playingFishX - W/2` 居中 | `playingFishY + 55` | **96 × 8**（由现状120×10略缩，更"贴身不抢眼"） | 复用 `_drawPixelHPBar`，已存在仅调宽 |
| HP 文字 | `playingFishX` 居中 | HP条下 +16 | 11px | 复用 `_drawPixelText`，去掉"鱼的体力"前缀只留「58/130」 |
| 拉力条 | `playingFishX - W/2` 居中 | `playingFishY + 78` | **96 × 6**（更窄，与 HP 区分主次） | 复用 `_drawPixelTensionBar`，锚点从 panel 改 fish |

**为什么是"上下两条小条贴鱼身"而不是合成一簇异形**：像素风下两条等宽水平条上下排是玩家最熟悉的读法（零学习成本），合成异形簇反而增加识别负担，违背"专注"目标。HP 在上（状态读取）、拉力在下（操作控制），符合"看一眼血、盯着拉"的视线分工。

---

## 2. 拉力的特殊性 ——【设计核心 / 老板痛点正解】

### 2.1 问题判断：拉力**不适合**完全跟随鱼身
老板担心点成立且关键。我的明确判断：

> **HP 适合完全跟随鱼身（它是"读取型"信息，瞄一眼即可）；拉力如果也完全跟随鱼身游动，会劣化操控手感——因为拉力是"高频盯着控制按住/松开"的操作型信息，鱼在动、条也跟着动，玩家眼睛要追着一个移动靶做精密操作，比固定位更累，与老板"更专注控制拉力"的目标背道而驰。**

简单平移拉力到鱼身 = 伤手感。所以拉力要做**双层拆分**。

### 2.2 拉力双层方案（推荐）：精确数值贴鱼身 + 危险态环境化

| 层 | 承载 | 形态 | 是否跟随鱼身 |
|---|---|---|---|
| **精确层** | 当前拉力值/黄金区/▲指针 | 鱼身下方那条缩窄拉力条（§1.2） | 跟随（让"看鱼"和"看拉力"在同一视觉焦点，不再左右扫） |
| **环境层** | 危险预警（将断/将松） | **复用已上线的全屏边缘光晕 + 鱼线变色**（无需新建） | 不跟随（铺满屏幕边缘，眼睛在任何位置都能感知） |

**关键利用现状**：`_renderPlaying()` 里**已经有现成的环境化危险提示**：
- 钓线变色：`_getFishingLineColor(this.tension)`（tension≥85 黄、≥95 红闪）
- 全屏边缘呼吸框：`lineWarnColor` red/blue 时 `strokeRect(3,3,cw-6,ch-6)`（L 约 2497）
- 屏幕中央「危险！」闪字 + tension≥85 全屏红闪警告牌

→ 这意味着**"危险感知"早已环境化，不依赖右下角面板**。玩家真正需要"低头看数字"的只有非危险区的精细微调，而那条缩窄拉力条贴在鱼身下方足够。

### 2.3 为什么这样能解决老板痛点
- **专注看鱼**：HP 与拉力都聚到鱼身焦点，视线不再在"鱼"和"右下角"之间来回扫（消除分流）。
- **专注控拉力**：危险时刻靠全屏边缘光晕/鱼线变色"环境化"接管——玩家进入危险时根本不需要找那条小条，整个屏幕边缘都在喊"松手/收线"；只有在安全区做精细微调时才瞄一眼鱼身下的小条。这比"盯一条跟着鱼游的小条"更稳、更专注。

## 3. 边缘裁切处理（鱼游到屏幕边缘/角落）

状态簇默认在鱼身**下方**，但鱼下潜到屏幕底部或贴边时会被裁切。处理规则（在绘制前算锚点）：

```
clusterY = playingFishY + 55
if (clusterY + 簇总高 > ch - safePadding)      // 鱼太靠下
    翻转到鱼身上方： clusterY = playingFishY - 簇总高 - 30
clusterX = clamp(playingFishX - W/2, safePadding, cw - W - safePadding)  // 左右贴边夹紧
```
- 竖向：下方放不下 → 翻转到鱼身上方（HP 与拉力整簇一起翻，保持相对顺序）。
- 横向：左右贴边时 X 夹紧到安全边距内（簇不再严格居中鱼身，但仍紧邻鱼身，不裁切）。
- `safePadding` 建议 12px。翻转用阈值 + 0.2s 缓动，避免鱼在边界反复横跳时簇频繁闪跳（防抖）。

## 4. HP 形态：环境暗示 + 鱼身下方极简条（组合）

老板要"专注看鱼的状态"。我的判断：**HP 值得做成"鱼身行为暗示（主）+ 鱼身下方极简数值条（辅）"的组合**，理由是这恰好把"看数字"升级成"看鱼"：

| 层 | 承载 | 实现 |
|---|---|---|
| **行为暗示层（主）** | 鱼挣扎力度随 HP 下降而减弱 | **已有数据可驱动**：搏斗 4 段节奏 fishPull 系数（试探×1.0→垂死×2.5）与 HP 挂钩；视觉上让鱼挣扎频率/幅度随 HP 衰减（HP 低 = 挣扎变弱变慢），玩家从"鱼还多猛"读出"还剩多少血" |
| **极简条层（辅）** | 精确 HP 数值 | 鱼身下方 96×8 缩窄条 + 「58/130」（§1.2），给想看准数的玩家兜底 |

- 这样 HP 的"专注看鱼"靠鱼本身的行为表达，数值条只是辅助确认，不抢视觉。
- ⚠ 实施提示：行为暗示层若要做"挣扎随 HP 减弱"的视觉调制，属于 `_renderFish` 动画参数微调，**非本次必做项**，可作为 P1 增强；本次 P0 先完成"去重 + 鱼身极简条"即可，行为暗示层标记为可选增强。

## 5. 主程改造锚点（基于真实代码，复用组件不重写）

### 5.1 删除右下角木牌面板（去重）
- 位置：`_renderPlaying()` L2533 起「右下角"鱼种"信息面板」整块。
- 改法：**保留鱼名 + 星级**（这是信息展示，不分散操作），**删除**该面板内的 HP 条（L2537-2539）与拉力条整段（L2542-2595，含黄金区高亮/▲指针/极限警告牌联动）。
  - 鱼名/星级可移到屏幕顶部或保留小面板仅显示名字星级（非 HP/拉力），由主程取轻量实现。
- ⚠ 注意：tension≥85 的「极限警告牌」全屏红闪（L 约 2598）属环境层，**保留**（它正是 §2.2 的环境化危险提示）。只删面板里的拉力**条**，不删全屏警告。

### 5.2 HP 条统一到鱼身下方（已存在，微调）
- 位置：L2467-2473 已有「鱼身体下方耐力条」。
- 改法：宽度 `fishHPBarW` 120→**96**；高度 10→**8**；文字去掉无（现状已是纯「X/Y」格式），保持 `playingFishY + 55` 锚点。基本沿用，仅缩尺寸让其更"贴身"。

### 5.3 拉力条搬到鱼身下方（唯一真正"搬家"的元素）
- 把 L2542-2595 这段拉力绘制逻辑（`_drawPixelTensionBar` + 黄金区高亮 + ▲指针 + 数字文字）**整体迁移**，锚点从 panel 坐标改为鱼身相对偏移：
  ```
  // 原（panel 坐标）
  const tensionBarW = hpBarW; const tensionBarX = hpBarX; const tensionBarY = panelY + 100;
  // 改为（鱼身相对偏移，与 §3 边缘翻转锚点统一）
  const tensionBarW = 96;
  const tensionBarX = Math.floor(playingFishX - tensionBarW / 2);  // 经 §3 clamp
  const tensionBarY = clusterY + 23;                                // HP条下方，经 §3 翻转
  ```
- 黄金区高亮、▲指针、tension≥85 条外闪烁框的内部逻辑**全部复用**，只是它们的基准坐标随 `tensionBarX/Y` 走。
- 拉力文字「鱼线拉力 X/Y」建议简化为「X/Y」（与 HP 一致省空间），或仅在非危险时显示。

### 5.4 复用组件清单（不重写）
- `_drawPixelHPBar(x, y, w, h, ratio, ...)` ✅
- `_drawPixelTensionBar(x, y, w, h, ratio, palette, ...)` ✅
- `_drawPixelText(...)` ✅
- `_getFishingLineColor` / `lineWarnColor` 全屏框 / 极限警告牌 ✅（环境层，保留）

## 7. 偏离字面的说明（已老板确认，2026-06-03）

> 老板字面是"HP 和拉力值**都显示在鱼的身体下方**"。本方案在两处对字面做了**有依据的偏离**，体验目标一致，已拍板：

1. **拉力没有 100% 只放鱼身下方** ✅ **老板拍板通过（2026-06-03）**：精确数值贴鱼身（符合字面），**危险预警走全屏边缘环境化**（不在鱼身下方）——复用现有边缘红框 + 鱼线变色 + tension≥85 全屏红闪。理由：拉力是高频操作信息，危险时若只靠鱼身下一条会让玩家在最紧张时低头找小条；环境化光晕让危险"满屏可感"，更服务"专注控拉力"的体验目标。**定稿，主程按 §2.2 + §4 实施。**
2. **HP 增加"鱼挣扎行为暗示"层**（P1 可选）：超出"显示在鱼身下方"的字面，但正是"专注看鱼"的最优解。本次 P0 不强制，先完成去重 + 鱼身极简条。
