# 美术资源交付规范 v1.2

> 版本：2026-05-28（PHASE 18 美术化阶段，v1.1 简化版）
> 适用：本项目所有 PNG 美术资源（NPC / 建筑 / 地块 / 道具 / 大背景）
> 基于实际代码量出，**所有数值均为像素**
>
> **v1.1 变更**：
> - NPC 尺寸统一为 **64×80**（不再按角色身高区分）
> - 素材内部控制（留白、居中精度、配色细节）改为"主美自行把控"，本规范只规定**程序对接的最小必要约束**（锚点、命名、格式）

## 1. NPC 类

### 1.1 标准尺寸

**所有 NPC 统一 PNG 尺寸 = 64×80**（含玩家阿明若后续美术化也同此规格）。

| 角色 | 文件名 | 尺寸 |
|---|---|---|
| 秀兰阿姨 | `npcs/xiulan.png` | **64×80** |
| 村长阿土伯 | `npcs/chief.png` | **64×80** |
| 林师傅 | `npcs/master_lin.png` | **64×80** |
| 小芳 | `npcs/xiaofang.png` | **64×80** |
| （阿明，玩家，参考） | 程序化 32×48 | — |

> 角色身高、体型、配饰是否撑满 64×80 由主美按角色性格自由发挥（如村长拐杖横展、林师傅高大、小芳矮小）。**程序侧只关心锚点（§1.2），不关心人物在 PNG 中的具体身高占比。**

### 1.2 锚点对齐（程序绘制依据，必须遵守）

程序按以下规则把 64×80 PNG 摆到地图上：

```
            ┌────────────────┐
            │                │  ← 头顶上方留白（主美自决）
            │                │
            │     人物       │
            │                │
            │                │
            │  ┌─脚底─┐      │  ← 脚底锚点（必须对齐）
PNG (64×80) └────────────────┘
                                ↑
              图片底边 = 角色脚底地面线
```

**仅 2 条对接铁律**：

| 锚点 | 程序如何处理 | 主美需保证 |
|---|---|---|
| **脚底贴 PNG 底边** | 代码以 `(npc.px-16, npc.py-32)` 绘制 64×80 图，使 PNG 底边正好落在 `py+48`（地面基线） | 角色脚底像素**贴近 PNG 底边**（不要悬空、不要被裁断） |
| **水平居中** | 代码使图片中心对齐瓦片中心 `npc.px+16` | 角色身体中心**位于 PNG 横向中线附近** |

> 如果角色脚底有阴影、特效、装饰元素，主美自行决定算不算"脚底"——只要游戏里**站在地面上不浮空、不陷地**即可。
>
> 横向居中也只需"看起来居中"即可，无需像素级精确。

### 1.3 朝向

**MVP 阶段：每个 NPC 仅提交正面图（下视角）一张 PNG**。

- 程序内 `dir` 参数（'down'/'up'/'left'/'right'）暂时被忽略
- 玩家在游戏中**总是绕到 NPC 正面对话**，背朝玩家的瞬间不易察觉
- 若主美愿意提供四向，可输出 4 张 PNG（同 64×80）按 `xiulan_down.png / xiulan_up.png / xiulan_left.png / xiulan_right.png` 命名，代码侧再支持

### 1.4 风格说明（参考向，不强制）

- 视角：2.5D 俯视（与现有像素 sprite 同视角）
- 背景：透明，不要白底/灰底
- 阴影：建议**不要**画地面阴影（代码可统一在脚底叠加，避免双层阴影）；如主美坚持画，告诉程序侧关掉代码阴影即可
- 配色：可参考现有像素 sprite（`js/render/characters.js`）和 §6 色板，但最终风格主美决定

## 2. 建筑类（v1.2 升级：物理/视觉占地解耦）

### 2.1 占地概念切分

| 概念 | 决定 | 谁定义 |
|---|---|---|
| **物理占地** | 碰撞、寻路、按 E 触发判定 | 关卡 / `village-scene.js`，固定瓦片格子 |
| **视觉占地** | PNG 在屏幕上的像素覆盖、向四周外溢量 | `js/render/buildings-art.js` 元数据表 |

> v1.2 起，**视觉占地不再强制 144×128**。主美按构图实际需要的比例交付，代码侧通过元数据表 `render.{w,h,anchorX,anchorY}` 适配；物理占地仍以瓦片为准。

### 2.2 当前各建筑契约

| 建筑 | 物理占地 | PNG 推荐尺寸 | 渲染目标尺寸 | 锚点偏移 (anchorX, anchorY) | 备注 |
|---|---|---|---|---|---|
| 村长家 (`chief_house`) | 2×1 | 144×128 | 144×128 | (0, 0) | 标准卡通像素风（基线） |
| 渔具店 (`fishing_shop`) | 2×1 | 任意（建议 ≤ 1024 边） | 156×128 | **(-12, 0)** | 横长构图，左侧棚棚溢出到广场（无碰撞冲突） |
| 阿明家 (`aming_house`) | 2×1 | 任意（建议 ≤ 1024 边） | 141×144 | **(0, -16)** | 屋顶上溢 16px、PNG 内含烟囱+灯笼柱+花丛，烟雾粒子由代码叠加 |
| 7-12 便利店 (`seven_eleven`) | 2×1 | 任意（建议 ≤ 1024 边） | 161×144 | **(0, -16)** | 招牌柱右溢 ~33px 进入村庄右边界空地，屋顶上溢 16px |

> **PNG 实际尺寸不限**（可 4× 高分辨率交付以备 Retina），代码用 `drawImage` 缩放到"渲染目标尺寸"。但 PNG 文件单边建议不超 1024px、单文件不超 200KB。

### 2.3 锚点与视觉外溢

- 渲染目标尺寸 `(w, h)` 决定 PNG 在屏幕上的像素覆盖
- 锚点偏移 `(anchorX, anchorY)` 相对"瓦片左上角 `(x, y)`"：
  - `(0, 0)` = PNG 左上角对齐瓦片左上角
  - `(-12, 0)` = PNG 向左溢出 12px（用于装饰性元素超出物理占地）
  - `(-8, -16)` = 屋顶顶尖向上溢 16px、左侧装饰外溢 8px
- **PNG 内部内容布局**（以 144×128 标准为例，非标准尺寸按比例换算）：
  ```
  ┌─────────────────┐
  │      屋顶       │ ← 上溢区域（瓦片之上，由 anchorY < 0 实现）
  ├─────────────────┤ ← y = 0 这条线对应瓦片左上 y
  │    墙身 / 门    │
  └─────────────────┘ ← 墙底
  ```
- 主美交付时，**人物入口/门**应位于图片底部偏中（让玩家可以走到门口）
- 同一行的建筑视觉底边应对齐（避免一栋"陷地"、一栋"悬空"）

### 2.4 添加新建筑的工作流

1. 主美按构图需要交付 PNG，**文件名全小写下划线**（如 `tavern.png`）
2. 程序在 `js/render/buildings-art.js` 的 `REGISTRY` 中加一行元数据：
   ```js
   tavern: {
     src: 'assets/images/buildings/tavern.png',
     tile: { cols: 3, rows: 2 },          // 物理占地（仅文档）
     render: { w: 240, h: 192, anchorX: 0, anchorY: -32 },
   }
   ```
3. 在 `village-scene.js` Layer3 中调用 `BuildingsArt.draw('tavern', ctx, x, y)`，加程序化兜底
4. PNG 加载失败 → 自动回退程序化绘制；PNG 就绪 → 立即生效，无需热更

### 2.3 动态元素（特别注意）

下列元素**当前由代码动态绘制**，主美**不要**画在 PNG 里：

| 建筑 | 动态元素 | 处理方式 |
|---|---|---|
| 阿明家 | 烟囱烟雾粒子（3 个圆向上飘） | 美术 PNG 留烟囱，烟雾代码叠加 |
| 阿明家 | 门缝暖光（夜晚渐亮） | 美术 PNG 不画门缝光，代码叠加 |
| 7-12 便利店 | 招牌霓虹灯闪烁 | 美术 PNG 画招牌底色，代码叠加发光 |
| 村长家 | 红灯笼一对 | 美术 PNG 可画也可不画；若画则关掉代码灯笼 |

**主美交付时附 README**：列出哪些动态元素已包含 / 待保留。

### 2.4 风格约束

- 同 NPC §1.4
- 颜色须与场景整体黄昏暖调（PALETTE.SKY_DUSK 系）和谐，**避免高饱和荧光色**
- 边缘可有轻微羽化但不超 2px

## 3. 地块（Tile）类

### 3.1 标准尺寸

每个地块 PNG **必须 64×64**，**无透明留白**（整图填满）。

| 地块类型 | 文件名 | 用途 | 是否可走 |
|---|---|---|---|
| 草地 | `tiles/grass.png` | 默认地面 | ✅ |
| 土路 | `tiles/path.png` | 村道 | ✅ |
| 石板路 | `tiles/stone_path.png` | 村中心广场 | ✅ |
| 水面 | `tiles/water.png` | 湖面（可平铺动画） | ❌ |
| 农田 | `tiles/farmland.png` | 菜园（已有程序化版本可参考） | ❌ |
| 花圃 | `tiles/flowerbed.png` | 花圃（已有程序化版本可参考） | ❌ |
| 树木 | `tiles/tree.png` | 树木（**树冠允许向上溢 32px**，PNG 64×96） | ❌ |
| 围墙 | `tiles/wall.png` | 石墙/木栅栏 | ❌ |

### 3.2 平铺要求

- **草地、水面、土路**必须**四边可无缝拼接**（左右上下边缘像素颜色连续）
- 树木可不无缝（独立物体）

### 3.3 数量

每种地块**至少出 1 张**，理想出 **3-4 张变体**（代码侧随机选取，避免大片重复观感）。命名 `grass_01.png / grass_02.png ...`

## 7. 交付检查清单（提交前自查）

每张 PNG 提交前请逐项确认：

- [ ] 文件名符合 §0.2（小写下划线，无中文）
- [ ] 格式 PNG，**RGBA**（带 alpha），背景透明
- [ ] 尺寸符合本文档对应类别的"标准 PNG 尺寸"（NPC 统一 64×80）
- [ ] **NPC**：脚底贴近 PNG 底边、人物大致水平居中（详见 §1.2）
- [ ] **建筑**：屋顶/墙体/门的位置满足 §2.2 的布局意图
- [ ] 文件大小 ≤ 50KB（背景类 ≤ 200KB）
- [ ] **不含**已由代码处理的动态元素（烟雾、灯光、水花……见 §2.3）
- [ ] 无白底/灰底（透明背景）

> 素材内部细节（留白多少、人物身高占比、配色精准度）由主美自行权衡，本规范不做硬性约束。

## 8. 验收流程

1. 主美按 §7 自查后交付到 `assets/images/{类别}/`
2. 程序侧用临时脚本扫描 PNG 透明边界（参考已有 `scripts/check_png_anchor.py`，待写）
3. 程序侧创建 `js/render/{类别}-{名字}.js` 模块加载贴图（参考 `js/render/npc-xiulan.js`）
4. 渲染入口（`village-scene.js` / `fishing-scene.js`）按"运行时检测 → 美术图优先 → 程序化兜底"模式接入
5. 出现比例/位置异常时，**先核对本规范 §1.2 / §2.2 锚点要求**
